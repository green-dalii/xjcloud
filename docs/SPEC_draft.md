二、第一性原理解：构建协作网络
2.1 核心洞察
乡村建设的瓶颈不是资源匮乏，而是协作成本太高。
当一个人想组织一场活动，他需要：找到场地、对接餐饮、协调交通、招募参与者、处理反馈——每个环节都是一道摩擦。
平台的价值是：把这些摩擦全部降低，让协作像呼吸一样自然。
2.2 三个层次的协作需求
层次
协作主体
核心需求
活动层
组织者 ↔ 参与者
活动发布 → 匹配 → 反馈
资源层
组织者 ↔ 服务方
场地、餐饮、交通的发现和对接
社区层
所有乡建人
经验分享、资源互换、联合行动
2.3 破局点
这三层协作需求的共同入口是：一场真实的活动。
- 有了活动，才有参与者的来到
- 有了参与者，才有服务方的对接
- 有了活动数据，才有社区的信任积累
所以，平台的破局点是：让一场活动从"难如登天"变成"一键生成"。

三、产品定位
3.1 平台定位
乡建协作基础设施
——不是内容平台，不是工具软件，而是让乡建人高效协作的数据驱动型网络
3.2 竞品重定义
竞品类型
我们与它们的本质差异

它们是"城里人看乡村"，我们是"乡建人协作网络"
微信群
它们是"人情驱动"，我们是"数据驱动+信任沉淀"
Eventbrite
它们是通用活动，我们是乡建垂直+AI+乡建DAO生态
飞书/钉钉
它们是"办公协作"，我们是"乡建场景协作"
3.3 乡建DAO的嵌入
乡建DAO不是平台的外挂，而是价值观骨架：
- 稻米积分：贯穿平台的激励货币（内容生产奖励、活动参与奖励、服务对接佣金）（重点是激励机制的设计）
- 节点治理：驿站的运营有社区参与，不是平台单方面管理
- 人情账本：信任可追溯、可视化
- 开源共建：技术架构对节点开放

5.2 活动方案AI生成（核心功能）
目标：组织者用3~5分钟回答问题，AI生成一份可直接执行的活动方案。
Prompt设计原则：
- 内置乡建场景知识（安全规范、非遗流程、研学特点）
- 输出结构化（时间表、材料清单、风险提示、参与者须知）
- 支持迭代修改（组织者可调整，AI学习偏好）
Human-in-the-Loop：
- 冷启动期：AI生成 → 人工审核 → 发布
- 成长期：AI生成 → 风险识别 → 疑似问题人工复核
- 成熟期：AI生成 → 自动发布 → 事后反馈修正

八、信任机制
8.1 为什么信任是核心问题
乡建场景的特殊性：
- 服务方大多是本地小农户/手艺人，没有标准化资质
- 参与者的期望和实际体验之间容易有落差
- 一次差的体验会摧毁长期的信任积累

11.1 黑客松交付目标
核心目标：演示一个完整的"AI生成活动方案 + 活动发布"闭环。
11.2 MVP功能范围
P0（今天必须完成）：
1. 组织者档案录入页（静态 + localStorage）
2. AI活动方案生成（调用Claude API，展示完整方案）
3. 活动发布页（展示生成的方案，含海报文案）
4. 活动列表页（展示已发布的活动）
5. 参与者报名入口（简化版表单）
P1（如果时间允许）：
6. 服务方主页（展示1~2个真实服务方信息）
7. UGC内容Feed（简化版）
P2（不做）：
- 支付功能
- 乡建DAO积分系统
- 推荐算法
11.3 演示脚本
开场（30秒）：
我们发现：乡建人有技能、有热情，但活动策划让他们望而却步。
我们做的是：用AI把策划门槛降到零。
演示（2分钟）：
1. 录入一位木艺传承人的信息（技能：木工；工具：锯子、砂纸、锤子；地点：龙潭村）
2. 回答3个问题（目标人群：亲子家庭；人数：15组；时长：半天）
3. 展示AI生成的完整方案：时间表、材料清单、安全提示、参与者须知、甚至如何让家长发朋友圈的建议
价值点（30秒）：
乡建人只需要填自己的特长，AI帮他想所有执行细节。
活动发布后，服务方（场地、餐饮）可以一键对接。
参与者报名数据帮助组织者优化下一场。
这就是数据飞轮。
